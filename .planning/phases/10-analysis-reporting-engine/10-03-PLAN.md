**analysis_prompts.py** - Prompt templates for Gemini synthesis:

Define prompt templates as string constants. Each prompt follows the grounding principle: "base your analysis ONLY on the provided facts and evidence."

1. `EXECUTIVE_SUMMARY_PROMPT`: Takes investigation objective and key facts. Instructs Gemini to write 1-2 paragraphs in analyst briefing voice summarizing findings. Include confidence level. Output: plain text string.

2. `KEY_JUDGMENTS_PROMPT`: Takes facts with classifications and verifications. Instructs to identify 3-10 key analytical judgments, each with confidence level (low/moderate/high), supporting fact IDs, and reasoning. Require JSON structured output matching KeyJudgment schema. Include examples of IC-style confidence language.

3. `ALTERNATIVE_HYPOTHESES_PROMPT`: Takes key judgments and facts. For each judgment with moderate or low confidence, generate 2-3 alternative interpretations. Require JSON structured output matching AlternativeHypothesis schema. Instruct to make alternatives structured, not wishy-washy.

4. `IMPLICATIONS_PROMPT`: Takes key judgments. Generate strategic implications and forward-looking forecasts. Output: JSON with `implications: list[str]` and `forecasts: list[str]`.

5. `SOURCE_ASSESSMENT_PROMPT`: Takes source inventory. Generate 1-paragraph assessment of source diversity, reliability, and gaps. Output: plain text string.

All prompts are concise (minimize tokens per CLAUDE.md cost optimization). Include explicit grounding instructions.

**synthesizer.py** - `Synthesizer` class:

Constructor takes:
- `config: AnalysisConfig`
- `llm_client: Any | None = None` (optional pre-configured Gemini client. If None, lazy-init from google.genai)

Uses structlog for logging.

Methods:

1. `async synthesize(snapshot: InvestigationSnapshot) -> AnalysisSynthesis`:
   Main orchestrator. Breaks synthesis into sections per RESEARCH.md Pattern 3 (avoid single monolithic LLM call):

   a. Prepare data context: serialize snapshot to concise text representation (facts as bullet points with IDs, classifications as tags, verification statuses). Keep under max_tokens_per_section.
   b. Generate executive summary: call LLM with EXECUTIVE_SUMMARY_PROMPT + data context
   c. Generate key judgments: call LLM with KEY_JUDGMENTS_PROMPT + data context -> parse JSON response into list[KeyJudgment]
   d. Generate alternative hypotheses: call LLM with ALTERNATIVE_HYPOTHESES_PROMPT + key judgments + facts -> parse into list[AlternativeHypothesis]
   e. Generate implications/forecasts: call LLM with IMPLICATIONS_PROMPT + key judgments -> parse JSON
   f. Generate source assessment: call LLM with SOURCE_ASSESSMENT_PROMPT + source inventory -> plain text
   g. Compute overall confidence from key judgments (weighted average of judgment confidences)
   h. Assemble AnalysisSynthesis from all parts

   Each LLM call wrapped in try/except. On failure, log error and use fallback (empty list for structured, "Analysis unavailable" for text). This ensures partial failures don't crash the entire synthesis.

2. `async _call_llm(prompt: str, structured: bool = False) -> str`:
   Call Gemini API via google-genai. Use `config.synthesis_model` and `config.temperature`. If structured=True, append "Respond with valid JSON only." Return raw response text.

3. `_prepare_facts_context(snapshot: InvestigationSnapshot) -> str`:
   Format facts as concise bullet points: `- [FACT-{id}] {claim_text} (confidence: {score}, status: {verification_status})`. Truncate to max_tokens_per_section.

4. `_parse_key_judgments(response: str) -> list[KeyJudgment]`:
   Parse JSON response. Handle malformed JSON gracefully (try json.loads, fall back to empty list with logged warning).

5. `_parse_alternative_hypotheses(response: str) -> list[AlternativeHypothesis]`:
   Same JSON parsing pattern.

6. `_compute_overall_confidence(judgments: list[KeyJudgment]) -> ConfidenceAssessment`:
   Average numeric confidence across judgments. Map to IC level: <0.4 = low, 0.4-0.7 = moderate, >0.7 = high.

**pattern_detector.py** - `PatternDetector` class:

No LLM dependency. Pure data analysis.

Constructor takes: nothing (stateless).

Methods:

1. `detect_patterns(snapshot: InvestigationSnapshot) -> dict[str, Any]`:
   Main method. Returns dict with:
   - `recurring_entities: list[dict]` - Entities appearing in 3+ facts, sorted by frequency
   - `source_clusters: list[dict]` - Groups of facts from same source type
   - `temporal_clusters: list[dict]` - Facts clustered by time proximity (within same day/week)
   - `escalation_indicators: list[dict]` - Sequences of facts about same entity showing escalation (increasing severity/impact)

2. `_find_recurring_entities(facts: list[dict]) -> list[dict]`:
   Extract entities from facts, count occurrences. Return entities with count >= 3.

3. `_find_temporal_clusters(facts: list[dict]) -> list[dict]`:
   Group facts by temporal proximity. Use temporal markers if available.

4. `_find_escalation_indicators(facts: list[dict], classifications: list[dict]) -> list[dict]`:
   For each entity appearing in multiple facts, check if classification severity increases over time (less_critical -> critical). Return detected escalation sequences.

**contradiction_analyzer.py** - `ContradictionAnalyzer` class:

No LLM dependency. Uses fact relationship data and verification results.

Constructor takes: nothing (stateless).

Methods:

1. `find_contradictions(snapshot: InvestigationSnapshot) -> list[ContradictionEntry]`:
   Identify contradictions from:
   a. Facts with explicit "contradicts" relationships (from fact.relationships)
   b. Verification results where status is REFUTED (the refuting evidence contradicts the original claim)
   c. Facts about the same entity with conflicting claims (same entity, opposing assertion_types)

   For each, create ContradictionEntry with:
   - description of the contradiction
   - fact_ids involved
   - resolution_status ("resolved" if one fact confirmed and other refuted, "unresolved" otherwise)

Follows VerificationPipeline/GraphPipeline lazy-init pattern exactly.

Constructor takes optional pre-configured components:
- `analysis_agent: AnalysisReportingAgent | None = None`
- `data_aggregator: DataAggregator | None = None`

- `config: AnalysisConfig | None = None`

Uses structlog for logging.

Methods:

1. `_get_aggregator() -> DataAggregator`:
   Lazy-init DataAggregator with shared stores and graph_pipeline.

2. `_get_agent() -> AnalysisReportingAgent`:
   Lazy-init AnalysisReportingAgent with config.

3. `async on_graph_ingested(investigation_id: str, ingestion_stats: dict[str, Any]) -> dict[str, Any]`:
   Event handler for graph.ingested events. Auto-triggers analysis if config.auto_generate_on_complete:
   - Aggregate data via DataAggregator
   - Run analysis via AnalysisReportingAgent.analyze()
   - Auto-generate Markdown report and save to ReportStore (if report_generator and report_store are available):
     ```python
     if self._report_generator is not None and self._report_store is not None:
         markdown = self._report_generator.generate_markdown(synthesis)
         await self._report_store.save_report(
             investigation_id=investigation_id,
             markdown_content=markdown,
             synthesis=synthesis,
         )
         self._log.info("report_auto_generated", investigation_id=investigation_id)
     ```
   - Log completion stats
   - Return analysis summary dict

4. `async run_analysis(investigation_id: str) -> AnalysisSynthesis`:
   Standalone mode:
   - Aggregate data
   - Run analysis
   - Auto-generate Markdown report and save (same pattern as on_graph_ingested)
   - Return AnalysisSynthesis

5. `def register_with_pipeline(investigation_pipeline: Any) -> None`:
   Register for `graph.ingested` events:
   ```python

       "graph.ingested",
       self.on_graph_ingested
   )
   ```
