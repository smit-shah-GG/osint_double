**fact_mapper.py** - `FactMapper` class:

Constructor takes `investigation_id: str`. Maintains internal state for entity resolution within the mapping session:
- `_entity_canonical_map: dict[str, str]` maps `canonical_name -> entity_node_id` for cross-fact entity dedup
- `_seen_sources: set[str]` for source node dedup

Methods:

1. `map_fact(fact: ExtractedFact, verification: VerificationResult | None = None, classification: FactClassification | None = None) -> tuple[list[GraphNode], list[GraphEdge]]`:
   - Creates Fact node: `id=fact.fact_id`, properties include claim_text, assertion_type, claim_type, content_hash, extraction_confidence, claim_clarity, temporal_value (if temporal marker present), temporal_precision, schema_version, investigation_id
   - Creates Entity nodes: For each entity in fact.entities, use `canonical or text` as the canonical name. Check `_entity_canonical_map` -- if canonical already seen, reuse the existing entity_node_id (entity resolution). If new, create Entity node with `entity_id="{investigation_id}:{canonical}"`, name, canonical, entity_type, cluster_id, investigation_id. Store aliases (all text variants seen) as list property.
   - Creates Source node: If fact.provenance exists, create Source node with source_id, source_type, investigation_id. Use `_seen_sources` to avoid duplicate source nodes.
   - Creates Investigation node: Ensure investigation node exists (one per mapper instance). Properties: investigation_id, created_at.
   - Creates edges:
     - `MENTIONS`: Fact -> Entity for each entity (properties: entity_marker like E1, E2)
     - `SOURCED_FROM`: Fact -> Source (properties: hop_count, attribution_phrase)
     - `PART_OF`: Fact -> Investigation
   - If verification is provided:
     - Add verification properties to Fact node: verification_status, final_confidence, confidence_boost, verified_at
     - Create `VERIFIED_BY` edge: Fact -> Investigation with verification metadata
   - If classification is provided:
     - Add classification properties to Fact node: impact_tier, dubious_flags
     - Optionally create Classification node linked via `HAS_CLASSIFICATION` edge
   - Returns (nodes, edges)

2. `map_facts_batch(facts: list[tuple[ExtractedFact, VerificationResult | None, FactClassification | None]]) -> tuple[list[GraphNode], list[GraphEdge]]`:
   Calls map_fact for each, aggregates results. Entity resolution works across the batch (shared _entity_canonical_map).

3. `_resolve_entity_id(canonical: str) -> str`:
   Check `_entity_canonical_map`. Return existing ID or create new one with format `"{investigation_id}:{canonical}"`.

Per CONTEXT.md: Entity resolution merges into canonical nodes with alias tracking. Single node per resolved entity. `aliases` property preserves all name variants. `resolution_confidence` score = 1.0 for exact canonical match, lower for fuzzy (but Phase 9 uses exact match only per RESEARCH.md open question 3).

**relationship_extractor.py** - `RelationshipExtractor` class:

Constructor takes `config: GraphConfig` (for llm_relationship_extraction flag).

Methods:

1. `extract_relationships(fact: ExtractedFact, verification: VerificationResult | None, existing_facts: list[ExtractedFact], investigation_id: str) -> list[GraphEdge]`:
   - Calls rule-based extraction first (always)
   - If `config.llm_relationship_extraction` is True, calls LLM extraction (expensive)
   - Deduplicates edges (same source+target+type = merge, keep higher weight)
   - Returns list of GraphEdge objects

2. `_extract_rule_based(fact, verification, existing_facts, investigation_id) -> list[GraphEdge]`:
   Rules derive from existing metadata -- these are cheap, no LLM calls:

   a. **CORROBORATES**: If verification result has supporting_evidence with 2+ independent sources, create CORROBORATES edges between fact and other facts that share the same claim_text or content_hash. Use verification.supporting_evidence authority scores for weight.
   b. **CONTRADICTS**: If verification result has refuting_evidence, create CONTRADICTS edges. If verification status is REFUTED, create edges between this fact and the refuting sources. Use FactRelationship objects from fact.relationships where type=="contradicts".
   c. **SUPERSEDES**: If verification status is SUPERSEDED and contradiction_type is "temporal", create SUPERSEDES edge from the newer fact to the older (linked via related_fact_id).
   d. **MENTIONS**: Already handled by FactMapper (Fact -> Entity). Skip here.
   e. **SOURCED_FROM**: Already handled by FactMapper. Skip here.
   f. **PART_OF**: Already handled by FactMapper. Skip here.
   g. **RELATED_TO**: Use fact.relationships where type=="supports" or type=="elaborates" to create RELATED_TO edges between facts.
   h. **LOCATED_AT**: If an entity has type LOCATION and another entity in same fact has type PERSON or ORGANIZATION, create LOCATED_AT edge from person/org entity to location entity. Weight based on claim_type (event = 0.7, state = 0.8).

   Use `compute_edge_weight()` from schema.py for weight calculation where authority_score and evidence_count are available.

3. `_extract_llm_based(fact, existing_facts, investigation_id) -> list[GraphEdge]`:
   For CAUSES, PRECEDES, ATTRIBUTED_TO edges that can't be derived from metadata alone:

   - Build a prompt with the fact's claim text and the claim texts of nearby facts (same entities, same investigation)
   - Use Gemini Flash (from existing llm_utils pattern or direct google-genai) to identify:
     - Causal relationships (CAUSES): "Event A led to Event B"
     - Temporal ordering (PRECEDES): "Event A happened before Event B"
     - Complex attribution (ATTRIBUTED_TO): "Claim was attributed to Entity X through Entity Y"
   - Parse structured JSON response from LLM
   - Convert to GraphEdge objects with lower base weight (0.4) since LLM-inferred

   If LLM call fails (rate limit, unavailable), log warning and return empty list (graceful degradation per existing patterns).

4. `extract_cross_investigation(fact: ExtractedFact, investigation_id: str, adapter: GraphAdapter) -> list[GraphEdge]`:
   - Query adapter for entities with same canonical name in OTHER investigations
   - If found, create RELATED_TO edges with `cross_investigation=True` on the edge
   - These are flagged only, not auto-trusted per CONTEXT.md
   - Per RESEARCH.md open question 3: exact canonical name match only
