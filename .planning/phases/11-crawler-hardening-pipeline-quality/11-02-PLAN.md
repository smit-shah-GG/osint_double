  In `_phase_crawl()`, after the article fetch loop (after `results = await asyncio.gather(*tasks, return_exceptions=True)` and the loop that collects successful results into `articles`), add RSS summary fallback logic.

  The current code at ~line 503-508:
  ```python
  articles: list[dict] = []
  tasks = [_bounded_fetch(u) for u in urls]
  results = await asyncio.gather(*tasks, return_exceptions=True)
  for r in results:
      if isinstance(r, dict):
          articles.append(r)
  ```

  Modify the fetch loop to track which entries failed and attempt RSS fallback:

  1. Change the fetch loop to pair URLs with their entry dicts. Instead of extracting `urls = [e["url"] for e in deduped]` and losing the entry association, iterate over deduped entries directly.

  2. After the fetch loop, for each failed fetch (result is None or exception), check if the original RSS entry has a `summary` or `description` field with >= 50 chars of content.

  3. If RSS summary is available:
     - Strip HTML from summary using `re.sub(r'<[^>]+>', '', text)` (RSS descriptions often contain HTML fragments per Pitfall 4 in RESEARCH.md)
     - Build an article dict with:
       ```python
       {
           "url": entry["url"],
           "title": entry.get("title", ""),
           "content": stripped_summary,
           "published_date": entry.get("published", ""),
           "source": {
               "name": domain,  # extracted from URL
               "type": "news_outlet",
               "authority_score": authority,  # use same domain->authority logic
           },
           "metadata": {
               "domain": domain,
               "content_source": "rss_summary",
               "content_length": len(stripped_summary),

  **In fact_schema.py:**

  1. Update the `Claim.claim_type` Literal (line 48) to include "statement":

     ```
     This is an additive change. Existing facts with other claim_types remain valid. No migration needed (per Pitfall 5 in RESEARCH.md).

  **In fact_extraction_agent.py:**

  2. Add a module-level normalization mapping (before the class definition):
     ```python
     # Normalize LLM-emitted claim_type values to valid schema values.
     # Models in the fallback chain (DeepSeek, Hermes, Nemotron) emit non-standard
     # values that Pydantic would either silently default or reject.
     _CLAIM_TYPE_NORMALIZE: dict[str, str] = {
         "statement": "statement",
         "action": "event",
         "fact": "statement",
         "opinion": "statement",
         "assertion": "statement",
         "observation": "state",
         "description": "state",
         "claim": "event",
         "announcement": "event",
         "plan": "planned",
         "forecast": "prediction",
         "analysis": "state",
     }
     _VALID_CLAIM_TYPES = {"event", "state", "relationship", "prediction", "planned", "statement"}
     ```

  3. Modify `_raw_to_extracted_fact()` to normalize claim_type before constructing the Claim object. Currently lines 580-584:
     ```python
     claim = Claim(
         text=claim_text,
         assertion_type=claim_data.get("assertion_type", "statement"),
         claim_type=claim_data.get("claim_type", "event"),
     )
     ```
     Change to:
     ```python
     raw_claim_type = claim_data.get("claim_type", "event")
     if isinstance(raw_claim_type, str):
         normalized_type = _CLAIM_TYPE_NORMALIZE.get(raw_claim_type.lower(), raw_claim_type.lower())
         if normalized_type not in _VALID_CLAIM_TYPES:
             normalized_type = "event"  # Final fallback for completely unknown values
     else:
         normalized_type = "event"

     claim = Claim(
         text=claim_text,
         assertion_type=claim_data.get("assertion_type", "statement"),
         claim_type=normalized_type,
     )
     ```

  Do NOT:
  - Remove any existing claim_type values from the Literal
  - Change assertion_type handling (that's separate and already works)
  - Add normalization for assertion_type (it already accepts "statement")
  </action>
